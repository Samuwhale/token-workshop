## Design Context

### Users
Figma UI/UX designers, design system creators and maintainers. They work inside Figma all day and expect plugin UI to feel native to the Figma environment. They are sophisticated users who understand design tokens conceptually but think in design terms, not developer terms.

### Brand Personality
**Precise, quiet, confident.** Like Figma itself — tools that stay out of the way and let the work speak. No hand-holding, no unnecessary flourish, no noise.

### Anti-References
- **NOT developer tooling**: No terminal aesthetics, JSON-heavy views, or dev-centric jargon
- **NOT enterprise SaaS**: No complex onboarding, permission gates, or corporate chrome
- **NOT toy / oversimplified**: Designers working with tokens are sophisticated — don't dumb it down

### Aesthetic Direction
Dark mode default (matching Figma's native theme), light mode support. UI should feel like a natural extension of Figma — same density, same interaction patterns, same visual language. Inter is the correct font choice (Figma's own typeface). The plugin lives in a constrained-width panel, so information density and vertical space efficiency matter.

### Accessibility
WCAG AA compliance. Standard contrast ratios, visible focus states, full keyboard navigation.

### Design Principles
1. **Feel native** — The plugin should be indistinguishable from a first-party Figma feature
2. **Respect expertise** — Designers using tokens are skilled; surface power without condescension
3. **Stay quiet** — No decorative UI, no unnecessary labels, no visual noise
4. **Show, don't tell** — Use color swatches, type previews, and spatial previews instead of raw values where possible
5. **Density over sprawl** — Pack information efficiently; vertical space is precious in a plugin panel
