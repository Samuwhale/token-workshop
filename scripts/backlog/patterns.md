# Codebase Patterns

Reusable patterns discovered during backlog work. This file is injected into every agent session.
Add new patterns here when discovered — keep entries general and reusable, not item-specific.

---

- **Pure-JS ZIP builder (no deps)**: For browser/plugin ZIP downloads, use stored (uncompressed) entries — no external library needed. Key: CRC-32 table with poly `0xEDB88320`, local header (30+name bytes, sig `0x04034b50`), central directory (46+name bytes, sig `0x02014b50`), EOCD (22 bytes, sig `0x06054b50`). All multi-byte fields are little-endian (`DataView.setUint16/32(pos, val, true)`). See `ExportPanel.tsx` `buildZipBlob` for a complete reference implementation.

- **Validation script**: `bash scripts/backlog/validate.sh` — runs 4 gates (tests, builds, lint, headless UI) after a `packages/core` bootstrap build. That bootstrap must happen before downstream tests and builds, otherwise fresh worktrees can fail with missing `@tokenmanager/core` package entry resolution in Vitest and `TS6305` in the server TypeScript build because `packages/server` references the composite core project. Do NOT use `npx pnpm …` or `npx turbo …` in worktrees — pnpm fails due to `virtualStoreDir` mismatch from the symlinked `node_modules`, and turbo can't find the pnpm binary. The validate script uses direct `pnpm` package commands plus `pnpm exec eslint`.
- **Shared-symlink worktree dependency rule**: When backlog worktrees bootstrap shared `node_modules`, treat them as read-only dependency environments. Do not run `pnpm install`, `pnpm add`, `npm install`, `yarn install`, `bun install`, or equivalent relinking commands from the temp worktree. If dependency links are stale or missing, repair them only from the main repo root by removing the poisoned package-local `node_modules` links and rerunning `pnpm install`.
- **ESLint config**: Root `eslint.config.js` (flat config). Runs via `npx pnpm lint`. Key rules: `react-hooks/rules-of-hooks` (error), `react-hooks/exhaustive-deps` (warn), `@typescript-eslint/no-unused-vars` (warn). Plugin sandbox files have `figma` and `__html__` globals.
- **Standalone UI harness**: `node packages/figma-plugin/standalone/serve.mjs` serves the built UI at `http://localhost:3200` outside of Figma with mocked postMessage. Useful for visual inspection. The `validate.mjs` script does a headless check for console errors.
- **Agent preview workflow**: `pnpm preview` is the repo-level way to inspect the plugin outside Figma. It builds the plugin, starts the plugin watch build, runs the local server against `demo/tokens`, and launches the standalone harness. If default ports are busy, use `pnpm preview -- --server-port 9410 --ui-port 3205`. The harness can boot the UI without Figma, but non-default server ports still need to be entered in the UI connection prompt because the plugin defaults to `http://localhost:9400`.
- **Quick-generator entry points**: For a lightweight generator flow that still needs parity with the full editor, reuse `useGeneratorDialog` with an `initialDraft` and call `handleQuickSave()` for the fast path. If the user escalates to advanced configuration, pass the current draft into `TokenGeneratorDialog` so the full form opens with the same type, target group, and config instead of resetting.
- **Token set file naming**: Files must end with `.tokens.json` (not just `.json`) for the TokenStore to discover them.
- **SVG chevron pattern**: Expand/collapse arrows use `<svg width="8" height="8" viewBox="0 0 8 8" fill="currentColor"><path d="M2 1l4 3-4 3V1z" /></svg>` with a `rotate-90` class toggled for direction. Never use `▶`/`▼` text characters.
- **SVG icon pattern (SyncPanel)**: `width="10" height="10" viewBox="0 0 24 24" fill="none" stroke="currentColor" strokeWidth="2" strokeLinecap="round" strokeLinejoin="round" aria-hidden="true"`. Checkmark: `M20 6L9 17l-5-5`. X: `M18 6L6 18M6 6l12 12`.
- **`</>` code icon**: `<path d="M8 6L4 12l4 6M16 6l4 6-4 6M13 4l-2 16"/>` with `strokeLinecap="round" strokeLinejoin="round"` on viewBox 0 0 24 24.
- **Context menu positioning**: Clamp raw mouse coords to `window.innerWidth/Height - menuWidth/Height` to prevent off-screen menus.
- **`setActiveTab` vs `setActiveTabState`**: Always use `setActiveTab` — it also persists to localStorage. Don't call `setActiveTabState` directly.
- **Alpha in color pickers**: HTML `<input type="color">` only supports 6-char `#RRGGBB`. Extract `hex.slice(7)` before the picker and re-append on change to preserve 8-char `#RRGGBBAA`.
- **ΔE thresholds**: <1 = "Exact", 1–5 = "Close", ≥5 = "Approximate".
- **Stale backlog references**: Some items reference code that no longer exists (e.g. "Generate Semantic Tokens" greyed menu items at App.tsx ~L687). Skip and pick a different item if the referenced code can't be found.
- **`syncSnapshot` comparison**: Use `stableStringify` (exported from `src/ui/shared/utils.ts`) for both snapshot storage (App.tsx) and comparison (TokenList.tsx). `JSON.stringify` key-order is non-deterministic for objects — never use it for token value comparison.
- **Token parent path computation**: Never use `path.lastIndexOf('.')` to find a segment boundary — segment names can contain literal dots (e.g., `"1.5"`). Use `nodeParentPath(node.path, node.name)` which is `path.slice(0, path.length - name.length - 1)`. Defined in `tokenListUtils.ts`.
- **Token path display**: When showing a full dotted path as text (tooltips, confirmations), use `formatDisplayPath(path, leafName)` to quote segments containing dots, e.g., `spacing."1.5"` instead of ambiguous `spacing.1.5`.
- **Inline color picker trigger**: Use a `<button>` with `onClick={() => inputRef.current?.click()}` next to a `<input type="color" ref={inputRef} className="sr-only" />`. Save on `onBlur` (fires when picker closes). Use `key={colorHex.slice(0,7)} defaultValue={...}` to reset the uncontrolled input when the external value changes.
- **Faint-always-visible button**: `opacity-40 group-hover:opacity-100 pointer-events-none group-hover:pointer-events-auto transition-opacity` — button is always dimly visible as a hint but only interactive when the row is hovered. Good for primary actions that should be discoverable without cluttering every row.
- **Sidebar layout alongside flex-col content**: Change the outer content wrapper from `flex flex-col` to `flex` (row), add sidebar as first child, wrap the remaining content (scroll div etc.) in a `flex-1 flex flex-col overflow-hidden` right-column div. The right-column takes `flex-1` and behaves identically to the original content area.
- **Set name validation**: Allow `/` for folder hierarchy with regex `^[a-zA-Z0-9_-]+(?:\/[a-zA-Z0-9_-]+)*$` — prevents leading/trailing slashes and double slashes.
- **Folder tree from flat names**: Parse `folder/set` names with `str.indexOf('/')` to build a `Map<folderName, FolderTreeNode>`. The `roots` array contains either plain `string` (unfoldered) or `FolderTreeNode` objects, preserving original order.

- **Token API response format**: `GET /api/tokens/:set` returns `{ set, tokens }` where `tokens` is a nested `TokenGroup` (DTCG format). `GET /api/tokens/:set/raw` also returns nested format. `getFlatTokensForSet` exists but is only used by `/api/docs` routes. `buildTree` in `useTokens.ts` is designed for nested format; flat input creates all tokens as top-level leaves.
- **Fastify route registration order**: Fixed routes (`/tokens/:set/raw`, `/tokens/:set` PUT) must be registered BEFORE the wildcard `/tokens/:set/*`. Fastify matches in registration order.

- **Graph/pipeline view pattern**: For "graph" features without a full node editor, use a pipeline visualization: stack generator cards with `[sourceToken] → [generatorName] → [targetGroup.* (N tokens)]` as a horizontal row. Template cards show `stages: string[]` rendered as pills with SVG chevron arrows between them. See `GraphPanel.tsx`.
- **Multi-step template application**: When a template creates multiple things (generator + semantic tokens), create the generator first, then POST semantic alias tokens using `{$type, $value: "{prefix.step}"}`. Best-effort the semantic tokens (log warn on failure, don't abort).
- **Exporting template data for command palette**: Export the template array from the panel component (`export const GRAPH_TEMPLATES`) so App.tsx can map over it for command palette entries without duplicating data.

- **`@tokenmanager/core` build**: `tsc` fails with pre-existing circular type errors in `types.ts`. A TypeScript watch process auto-compiles dist files when src changes. Editing core src is sufficient — dist will be regenerated automatically. Never try to manually patch dist JS files.
- **`flattenTokenGroup` in core**: `flattenTokenGroup(group, prefix?, parentType?): Map<string, DTCGToken>` in `dtcg-types.ts` is the canonical DTCG flattener. It handles `$type` inheritance from parent groups. Use it instead of local flatten implementations. To get `TokenMapEntry` records, iterate the Map: `for (const [path, token] of flattenTokenGroup(group)) { out[path] = { $value: token.$value, $type: token.$type || 'unknown' }; }`.

- **Hook declaration order (TDZ)**: In large React components, always declare `useState`/`useRef` variables **before** any `useMemo`, `useCallback`, or plain `const` expressions that reference them. `const` and `let` are subject to the Temporal Dead Zone — accessing them before their declaration line throws `ReferenceError: Cannot access 'X' before initialization` at runtime. TypeScript does **not** catch this because it only checks that the variable is declared *somewhere* in scope, not that it precedes every use. The bug is invisible in source but manifests as a cryptic TDZ crash when the component first renders. Rule: all `useState`/`useRef` calls that feed a `useMemo`/`useCallback`/derived-const must appear above that expression in the function body.

- **No re-exports from `@tokenmanager/core` in UI utility files**: Never re-export symbols from `@tokenmanager/core` through shared utility files like `colorUtils.ts`. The plugin UI is bundled into a single HTML file via `vite-plugin-singlefile`, and re-export chains create module initialization ordering issues that manifest as minified TDZ errors (e.g., `Cannot access 'Ao' before initialization`). Instead, have each component import directly from `@tokenmanager/core`. Keep shared utility files for locally-defined helpers only.

- **`.tsx` for any file with JSX**: Even "helper" or "utility" files that return JSX (e.g., `highlightMatch` returning `<mark>` elements) must use `.tsx` extension. The Vite/esbuild pipeline does not process JSX in `.ts` files, causing "Unexpected token" build errors on `<>` fragments.

- **TokenList component structure**: `TokenList.tsx` is the orchestrator (~2660 lines) that manages all state and callbacks. Sub-components: `TokenTreeNode.tsx` (tree row renderer, 1341 lines), `TokenListModals.tsx` (all modal dialogs, 658 lines), `tokenListTypes.ts` (shared types/constants), `tokenListHelpers.tsx` (pure utility functions). The `TokenTreeNodeProps` interface is in `tokenListTypes.ts`.
- **Hover action buttons without layout shift**: For action buttons that appear on row hover, use `absolute right-1 top-0 bottom-0 flex items-center` on a wrapper div instead of making buttons inline flex children with `opacity-0`. Inline opacity toggles keep the element in flow, causing layout shift. Absolute positioning removes them from flow entirely. The parent row needs `relative`. Apply `pointer-events-none group-hover:pointer-events-auto` on the wrapper, and `opacity-0 group-hover:opacity-100 transition-opacity` on individual buttons.

- **Two-tier tab navigation**: Top-level tabs use `TOP_TABS` array with `{ id, label, subTabs }`. State: `activeTopTab` + `activeSubTab`, persisted per-top-tab in localStorage via `SUB_TAB_STORAGE` map. `navigateTo(topTab, subTab?)` sets both atomically and clears overflow. When `subTab` is omitted, restores last-used sub-tab from localStorage or falls back to `DEFAULT_SUB_TABS[topTab]`. Sub-tab row renders conditionally when the active top-tab has >1 sub-tab. Old `activeTab`/`setActiveTab` kept but unused for migration safety.
- **`setActiveTab` vs `navigateTo`**: Always use `navigateTo(topTab, subTab)` for navigation. The old `setActiveTab` is retained for backward compat but should not be called for new code. `navigateTo` also calls `setOverflowPanel(null)` internally.

- **Per-theme-option token resolution**: To resolve tokens for a specific theme option, collect non-themed tokens as base layer, apply `source` sets, then `enabled` sets (overrides). Use `resolveAllAliases` on the merged result. This mirrors `useThemeSwitcher`'s logic. Key data: `dimensions` (from `/api/themes`), `unthemedAllTokensFlat` (raw flat tokens), `pathToSet` (token path → set name mapping).
- **Set routing for theme-aware edits**: When editing a token value for a specific theme option, prefer the first `enabled` set that already contains the token (existing override), fall back to the first `enabled` set (new override), then `source` sets. This ensures edits land in the right override file.

- **Stable callback ref for event effects**: When a `useEffect` registers a global event listener (keydown, resize, etc.) that calls a handler depending on many state variables, use a mutable ref: declare `const fnRef = useRef<(...) => void>(() => {})`, assign `fnRef.current = fn` after the function definition on every render, and call `fnRef.current(...)` inside the effect. This avoids listing all transitive state deps and prevents re-registering the listener on every state change.

- **Promise-chain mutex for async file I/O**: To prevent load-modify-save races in Node.js async handlers, use a promise-chain lock: `let lockChain = Promise.resolve()`. In `withLock(fn)`: `const next = lockChain.then(() => fn()); lockChain = next.catch(() => {});  return next;`. Each caller chains behind the previous, serializing access. No external deps needed. Throw errors with `statusCode` property from inside the lock callback for HTTP error propagation.

- **Promise-chain mutex for async file I/O**: To prevent load-modify-save races in Node.js async handlers, use a promise-chain lock: `let lockChain = Promise.resolve()`. In `withLock(fn)`: `const next = lockChain.then(() => fn()); lockChain = next.catch(() => {});  return next;`. Each caller chains behind the previous, serializing access. No external deps needed. Throw errors with `statusCode` property from inside the lock callback for HTTP error propagation.

- **Server-side operation log pattern**: For undo/rollback of server mutations, use before/after snapshots captured around each destructive route handler. `OperationLog` service in `packages/server/src/services/operation-log.ts` stores entries in a ring buffer. Snapshot helpers (`snapshotPaths`, `snapshotSet`, `snapshotGroup`) are exported from the same file. `TokenStore.restoreSnapshot()` handles batch token restoration. Routes: `GET /api/operations?limit=N`, `POST /api/operations/:id/rollback`.

- **Extracting custom hooks from large components**: When extracting state groups into custom hooks, place the hook call after all its memoized dependencies (e.g., `siblingOrderMap`) are computed. Destructure all return values at the call site so existing JSX and callbacks can reference them unchanged. Move any `useCallback` that references destructured hook state to after the hook call to avoid TDZ. Replace inline JSX event handlers that set multiple related states (e.g., `() => { setA(null); setB(null); }`) with dedicated handler functions in the hook that encapsulate the cross-state interactions.

- **TokenTreeContext for shared tree state**: `TokenTreeContext.tsx` provides `useTokenTree()` hook. All shared callbacks and data (36+ values) flow through context; only per-node props (`node`, `depth`, `isSelected`, `lintViolations`, `skipChildren`, `showFullPath`, `isPinned`, `chainExpanded`, `onMoveUp`, `onMoveDown`, `multiModeValues`) are passed as props. The context value is memoized in `TokenList.tsx` via `useMemo`. Both pinned and virtual-scroll sections share the same `<TokenTreeProvider>`.

- **SVG node-graph editor pattern**: For interactive node editors, use a single SVG with `<g transform="translate(pan.x, pan.y) scale(zoom)">` as the pan/zoom layer. Nodes are `<g transform="translate(node.x, node.y)">` groups inside it. Edges are bezier curves via `M x1,y1 C cx1,y1 cx2,y2 x2,y2`. Use `<foreignObject>` for HTML form controls inside nodes. Port wiring: track `fromDirection` ('in'|'out') so the bezier always flows correctly. Screen-to-graph conversion: `graphX = (screenX - rect.left - pan.x) / zoom`. Zoom toward cursor: `newPan = cursor - (cursor - oldPan) * (newZoom / oldZoom)`.

- **Wide-gamut color support**: CSS Color 4 strings (`oklch()`, `oklab()`, `color(display-p3 ...)`) can be stored as DTCG token `$value` strings. Use `swatchBgColor()` from `colorUtils.ts` for all `backgroundColor` style props (replaces `.slice(0,7)`). Use `isWideGamutColor()` to detect out-of-gamut colors and `getSrgbFallback()` for the gamut-mapped hex. The `parseAnyColor` / `serializeColor` / `toOklch` / `toSrgb` functions in `@tokenmanager/core` handle all conversions. `GamutIndicator.tsx` renders a "P3" badge. Plugin sandbox has its own OKLab math in `plugin/colorUtils.ts` since it can't import from core.

- **`$extensions.tokenmanager.extends` for token inheritance**: Composite tokens (typography, shadow, border, transition, composition) can inherit from a base token. Stored at `$extensions.tokenmanager.extends` (string path). `COMPOSITE_TOKEN_TYPES` set in `constants.ts` identifies eligible types. `TokenResolver.getExtendsPath(token)` extracts the path. The resolver shallow-merges `{ ...baseValue, ...localValue }` during DFS resolution. Extends targets are included in the dependency graph for cycle detection. UI editors accept `baseValue` prop and show inherited/overridden indicators per property.

- **SVG bezier curve editor pattern**: For interactive cubic bezier editors, use `setPointerCapture` on control point circles. Convert screen coords to normalized [0,1] via `fromSvg(clientX - rect.left, clientY - rect.top)`. Solve x→y with Newton-Raphson (8 iterations, `|err| < 1e-7`) plus bisection fallback (20 iterations) when Newton goes out of range. Curve path: `M sx0,sy0 C scx1,scy1 scx2,scy2 sx1,sy1`. Control point handles as dashed lines from anchors `(0,0)` and `(1,1)`.
- **Reverse-solving ratio from dragged value**: For type/spacing scales with `value = base * ratio^exponent`, dragging a step to a desired value gives `newRatio = (desiredValue / base)^(1/exponent)`. The base step (exponent=0) cannot be used to solve ratio. Snap to named values with a threshold (~0.015) for better UX.

- **Vite resolve alias for workspace packages**: The figma-plugin `build.mjs` has `resolve.alias['@tokenmanager/core']` pointing to `packages/core/src/index.ts` (TypeScript source). This eliminates the need for a separate core build step — Vite compiles core TS directly. In worktrees this is essential since `node_modules/@tokenmanager/core` symlinks to the original project's dist, not the worktree's modified source.

- **Typed HTTP errors for server routes**: `packages/server/src/errors.ts` exports `NotFoundError(msg)` (404), `ConflictError(msg)` (409), `BadRequestError(msg)` (400), and `handleRouteError(reply, err, fallbackMsg?)`. Services throw typed errors; routes catch with `handleRouteError`. Also supports legacy `Object.assign(new Error(), { statusCode })` pattern. Always use `.js` extension in imports.

- **Structural rollback steps pattern**: For operations that change non-token state (sets, themes, resolvers), add `rollbackSteps: RollbackStep[]` to the operation log entry. Steps are executed before token snapshot restoration during rollback. Inverse steps are computed before execution (capturing current state) and stored on the rollback's own operation entry for redo support. Available step actions: `create-set`, `delete-set`, `rename-set`, `reorder-sets`, `write-themes`, `write-resolver`, `delete-resolver`. The `RollbackContext` passed to `rollback()` provides `tokenStore`, `resolverStore?`, and `generatorService?`.

- **Cross-component settings sync via custom events**: `SettingsPanel.tsx` exports `dispatchSettingsChanged(key)` and `useSettingsListener(key)`. When a setting is changed, call `dispatchSettingsChanged(STORAGE_KEYS.X)` after writing to localStorage. Consuming components call `const rev = useSettingsListener(STORAGE_KEYS.X)` and re-read from localStorage in a `useEffect` triggered by `rev`. Skip when `rev === 0` (initial mount). This avoids prop-drilling or context refactoring for settings that multiple unrelated components read from localStorage.

- **Atomic file writes for watched directories**: When writing JSON files in directories monitored by chokidar, always write to a `.tmp` file first, then `fs.rename` to the final path. `fs.rename` is atomic on the same filesystem, so watchers never see partial content. Place the write guard (`_startWriteGuard`) immediately before the rename, not before the `writeFile`. This pattern is used in both `token-store.ts` and `resolver-store.ts`.

- **Tab-to-timeline migration**: When converting a multi-tab panel into a unified chronological list, keep each tab's detail/sub-view component intact. Add `initialSelectedId` + `onBack` props to jump directly into the sub-view (bypassing the list view) and skip redundant list fetches with a `skipListFetch` prop. The parent panel owns the selected-item state and calls `onBack` to return to the timeline. Merge all entry types into a discriminated union sorted by timestamp, render type pills to distinguish sources visually.

- **Imperative handle for cross-component actions**: To let a parent (e.g., App.tsx command palette) trigger internal actions in a deeply-stateful child (e.g., TokenList), add `interface FooImperativeHandle { doX: () => void; doY: (arg: string) => void; }` to the shared types file, add `fooHandle?: React.MutableRefObject<FooImperativeHandle | null>` to the child's props, and populate it in the child via `useEffect(() => { if (!fooHandle) return; fooHandle.current = { doX: () => setX(true), doY: setY }; return () => { fooHandle.current = null; }; }, [fooHandle])`. `useState` setters are stable, so no extra deps needed. The parent creates `const ref = useRef<FooImperativeHandle | null>(null)` and calls `ref.current?.doX()`. Much cleaner than lifting state or trigger-counter props.

- **vitest.config.ts for workspace alias**: Add `packages/figma-plugin/vitest.config.ts` with `resolve.alias['@tokenmanager/core']` pointing to `packages/core/src/index.ts` to fix vitest's inability to resolve workspace packages in the test environment (Vite builds use `build.mjs` aliases, but vitest doesn't inherit them). Without this, any test that imports a module that transitively imports `@tokenmanager/core` will fail with "Cannot find package".

- **Shared comparison utilities**: `packages/figma-plugin/src/ui/shared/comparisonUtils.ts` exports `resolveThemeOption(option, allTokensFlat, pathToSet, themedSets?)`, `exportCsvFile(filename, rows)`, and `copyToClipboard(text, onSuccess?, onError?)`. Use these instead of local implementations in comparison components. Pass optional `themedSets` to `resolveThemeOption` to include unthemed tokens as a base layer before source/enabled sets.

- **Plugin sync snapshot round-trip for revert**: Figma VariableValue, Paint[], Effect[], and text style fields are all JSON-serializable via postMessage. Capture pre-sync state in the plugin (before mutations), include in the success response message, store in UI state, and send back as payload for a `revert-*` message. The plugin then restores the snapshot. This avoids any server involvement. Key: snapshot Map entries as `Record<id, state>` for serialization, track created IDs separately for deletion on revert.

- **`VariableSyncSubPanel` / `StyleSyncSubPanel` location**: Variable sync diff UI is in `VariableSyncSubPanel.tsx` (rendered inside the "Figma Variables" accordion in PublishPanel). Style sync diff UI is rendered inline within the "Figma Styles" accordion in PublishPanel (the `StyleSyncSubPanel` component exists but is not used — inline code is canonical). Git UI uses `GitSubPanel`.

- **Feature-area context provider pattern**: To isolate re-render cascades in a large orchestrator component (like App.tsx), extract groups of related hooks into context providers: `ConnectionProvider` (server/sync), `TokenDataProvider` (token sets/flat maps/generators), `ThemeProvider` (theme switching/resolver), `InspectProvider` (selection/heatmap/usage counts). Each provider calls `useMemo` for its context value object to prevent spurious consumer re-renders. Inner providers call `useXxxContext()` from outer providers instead of accepting props. Provider nesting: outermost is ConnectionProvider (no deps), then TokenDataProvider (calls useConnectionContext), then ThemeProvider (calls both), then InspectProvider (standalone). `main.tsx` or entry point stacks them as a nested tree around the App component. App.tsx calls `useXxxContext()` hooks at the top and destructures the needed values. The four context files live in `packages/figma-plugin/src/ui/contexts/`.

- **Generic sync hook pattern (`useSyncEntity`)**: Split hook config into two parts: (1) `SyncMessages` — static message types + `extractResponse` functions as **module-level constants** (required stable for `useFigmaMessage` dep array); (2) dynamic config — entity-specific build functions stored in a ref (`configRef.current = config`) so they stay current without re-registering anything. Returns neutral property names (`rows`, `dirs`, `loading`, etc.). Callers call `useSyncEntity<RowType, SnapshotType>(serverUrl, activeSet, connected, MESSAGES_CONST, dynamicConfig)`.

- **Generic sync panel pattern (`SyncSubPanel`)**: Parameterize diff UI by text props (`description`, `sectionLabel`, `previewAction`, `applyAction`, `inSyncMessage`, `notCheckedMessage: React.ReactNode`, `revertDescription`) plus a `sync` prop typed as the return of `useSyncEntity`. The `notCheckedMessage` prop accepts `ReactNode` to support mixed HTML/text (e.g. `<>Click <strong>Compare</strong> to see...</>`) without needing a render prop.

- **Generator input primitives**: `generatorShared.tsx` exports `CompactColorInput` (hex color swatch + text input), `UnitToggle<T extends string>` (typed pill buttons), and `CompactDimensionInput` (number + UnitToggle combined). Use these in all generator config editors instead of inline implementations. `CompactColorInput` takes `value: string` (hex), `onChange: (hex) => void`, and optional `aria-label`.
- **Collapsible component**: `Collapsible.tsx` is a controlled (open + onToggle props) chevron-button toggle. The `label` prop is `ReactNode` so extra spans (e.g. active-type indicators) can be embedded inline. Children render directly without a wrapper div, so callers control content margins.
- **TOKEN_TYPE_CATEGORIES**: `src/ui/shared/tokenTypeCategories.ts` exports the DTCG token type optgroup structure. Use for any `<select>` or type picker that shows DTCG types. Also exports `ALL_TOKEN_TYPES: string[]` for flat access.
- **TYPE_LABELS (generator)**: `generatorUtils.ts` exports `TYPE_LABELS: Record<GeneratorType, string>` — the canonical human-readable labels for generator types. Never define this inline; import from generatorUtils in all generator-related files.

- **Modal context pattern for prop-drilling elimination**: When a component receives 50+ individual props from a parent, create a dedicated context: `interface ModalsState = ExistingPropsInterface`, `createContext<ModalsState | null>(null)`, `<ProviderComponent value={useMemo(() => ({...all props}), [deps])}>`, and `useModalsContext()` hook inside the consumer. The consumer function signature drops all props. The `useMemo` on the parent side is the performance key — modal-only renders don't cascade from unrelated parent state changes (search, scroll, expansions). See `TokenListModalsContext.tsx` + `TokenListModals.tsx`.

- **Circular-import prevention for context extraction**: When extracting a context from App.tsx, types/constants that both App.tsx and the new context need must be extracted to a `shared/xxxTypes.ts` file first. Without this, App.tsx importing the context AND the context importing types from App.tsx creates a circular dependency. Pattern: move types → create context importing from shared file → update App.tsx to import from shared file.

- **Late-bound callback ref in context providers**: When a provider needs a callback (e.g., onAliasNotFound) that depends on state from a child component (App.tsx's useToastStack), use a ref: `const callbackRef = useRef(() => {})`. Expose a stable `setHandler(fn)` via context. The child component (App.tsx) calls `setHandler(fn)` in a `useEffect` after the callback source is available. Since stable useCallback functions (useToastStack's pushError) don't change, an empty-dep effect is safe.

- **Shared field validation component**: `FieldMessage.tsx` in `packages/figma-plugin/src/ui/shared/` renders error/warning/info text below a form field. Priority: error (red, role="alert") > warning (amber-500) > info (tertiary). `fieldBorderClass(hasError?, hasWarning?)` in `editorClasses.ts` returns the correct Tailwind border classes (error=red/red-focus, warning=amber-400, normal=border/accent-focus).
