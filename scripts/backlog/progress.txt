## 2026-03-27 - Undo doesn't cover set-level operations
- Added undo/redo support for all set-level operations: rename, delete, create, duplicate, merge, and split.
- Rename: captures old/new names, restore renames back, redo renames forward.
- Delete: snapshots tokens before deleting, restore recreates set via POST, redo deletes again.
- Create: restore deletes the created set, redo recreates it (empty).
- Duplicate: captures duplicated tokens, restore deletes the copy, redo recreates it.
- Merge: snapshots target tokens before merge (pre-merge state), restore uses `PUT /api/tokens/:set` to replace all tokens. No redo (complex multi-step).
- Split: tracks which sets were created, restore deletes them and optionally recreates original if it was deleted. No redo.
- Files changed: `packages/figma-plugin/src/ui/App.tsx` ~L542 (rename), ~L654 (delete), ~L632 (create), ~L694 (duplicate), ~L780 (merge), ~L839 (split)
- **Learnings for future iterations:**
  - `PUT /api/tokens/:set` (replaceSetTokens) is the correct endpoint for bulk-restoring a set's token state — avoids needing to delete+recreate the set and losing theme references.
  - For undo of multi-step operations (merge, split), skip redo if the reverse would require re-fetching complex state — partial undo-only slots are supported by the UndoSlot interface (`redo` is optional).
  - Always capture necessary state (serverUrl, tokens) into local consts before pushing the undo slot, since the closure runs asynchronously after the handler completes.

## 2026-03-27 - Connection-drop mid-operation has no recovery
- Added `markDisconnected()` to `useServerConnection.ts`: immediately flips `connected` to `false`, fires the disconnect AbortController, and triggers an immediate health re-check — so the existing reconnect banner appears at once rather than waiting up to 5 s for the next health poll.
- Added `getDisconnectSignal()` getter: returns the current `AbortController.signal` for the active connection; callers combine it with `AbortSignal.any([AbortSignal.timeout(5000), getDisconnectSignal()])` so in-flight fetches are cancelled the moment the health poll detects a drop.

- Updated `useTokens.ts`: accepts optional `onNetworkError` and `getDisconnectSignal` params; `refreshTokens` uses `AbortSignal.any` with the disconnect signal and calls `onNetworkError()` (→ `markDisconnected`) on `TypeError` / "Failed to fetch".
- Wired `markDisconnected` + `getDisconnectSignal` into key App.tsx mutation operations (rename set, create set, delete set, duplicate set) and background fetches (git sync status, `fetchAllTokensFlatWithSets`). Each adds `AbortSignal.any([timeout, getDisconnectSignal()])` to the fetch and calls `markDisconnected()` in the catch block on network errors.
- The existing `!connected` banner (with Retry + Change URL links) serves as the reconnect prompt — no new UI needed; the improvement is that it now appears immediately on mid-operation failure instead of on the next health-poll cycle.
- Files changed: `packages/figma-plugin/src/ui/hooks/useServerConnection.ts` (full rewrite), `packages/figma-plugin/src/ui/hooks/useTokens.ts` ~L50 (refreshTokens), `packages/figma-plugin/src/ui/App.tsx` ~L71 (useSyncBindings), ~L214 (hook call), ~L273 (git sync fetch), ~L394 (fetchAllTokensFlatWithSets), ~L552 (handleRenameConfirm), ~L620 (handleCreateSet), ~L641 (handleDeleteSet), ~L665 (handleDuplicateSet)
- **Learnings for future iterations:**
  - `AbortSignal.any([AbortSignal.timeout(N), disconnectSignal])` is the correct pattern to abort on either timeout OR explicit disconnect — already used in `useLint.ts`, now applied consistently.
  - Reading `disconnectControllerRef.current.signal` at the CALL SITE (via a `getDisconnectSignal()` getter) rather than capturing it in a closure avoids stale-signal issues between React renders.
  - The 5-second health-check poll means users can be stuck for up to 5 s before the disconnect banner appears. `markDisconnected()` bridges this gap for synchronous error paths; the getter + `AbortSignal.any` bridges it for the fetch timeout path.

## 2026-03-27 - Partial sync failure has no rollback — if sync fails partway through a batch, already-applied changes persist with no way to revert
- Added snapshot-based rollback to both sync operations in controller.ts:
  1. `applyVariables()`: Before the token loop, records `{modeId: value}` snapshots of all pre-existing variables and tracks IDs of any newly created variables/collections. On error, restores original variable values and removes newly created variables/collections; reports `rolledBack: true/false` in the `apply-variables-error` message.
  2. `syncBindings()`: Added `captureNodeProps()` / `restoreNodeProps()` helpers. Before processing each node, snapshots the property values that its bindings will modify (fills, strokes, padding, etc.) via JSON round-trip. Wrapped the batch loop in a try/catch; on unexpected outer error, restores all node snapshots and emits `sync-complete` with `rolledBack` status + Figma notification.
- Updated `SyncPanel.tsx` error handler to append "(changes rolled back)" or "(rollback failed — some changes may persist)" to the displayed error message.
- Files changed: `packages/figma-plugin/src/plugin/controller.ts` ~L128 (applyVariables rollback), ~L1289 (captureNodeProps/restoreNodeProps helpers + syncBindings rollback)
- Files changed: `packages/figma-plugin/src/ui/components/SyncPanel.tsx` ~L214 (rollback note in error)
- **Learnings for future iterations:**
  - Figma variables expose `variable.valuesByMode` as a `Record<modeId, VariableValue>` — spread-copy it for a shallow snapshot that's sufficient for rollback since VariableValue is a scalar (number/string/boolean/RGBA).
  - For node property snapshots, JSON.parse(JSON.stringify(val)) is the safest deep-copy for Paint arrays; direct spread misses nested objects in gradient stops.
  - Rollback code should be in the catch block OUTSIDE the main try, with its own inner try/catch to prevent rollback errors from hiding the original error.
  - New collections should only be removed on rollback if they're empty (newly created variables may have already been deleted).

- Added Cancel button that appears next to the Syncing… button during an active variable sync. Clicking it sets `syncCancelledRef`, aborts the pull-phase fetch, and rejects any pending Figma apply promise — stopping the sync at the earliest safe checkpoint without showing an error to the user.
- Added two refs: `syncCancelledRef` (boolean flag checked between phases) and `syncFetchAbortRef` (AbortController passed to the pull fetch). Added `cancelVarSync` callback that sets the flag, aborts the fetch, and rejects pending Figma promises.

- **Learnings for future iterations:**
  - Two-phase async operations (message-passing + fetch) need two cancellation mechanisms: reject the pending promise map for the message-passing phase, and abort an AbortController for the fetch phase.
  - Check `syncCancelledRef.current` between phases (not inside the promise) to avoid race conditions.
  - Suppress error display on cancellation with `if (!syncCancelledRef.current)` in the catch block — cancellation is intentional, not an error.
