## 2026-04-05 - Token rename does not propagate to Figma variable names
- Implemented rename propagation: when tokens are renamed on the server, the next variable sync in Figma renames the existing Figma variable to the new name instead of creating a new one and orphaning the old one. This preserves the Figma variable ID and all node bindings referencing it.
- Flow: rename routes record `pathRenames` in the operation log → server exposes `/api/operations/path-renames` → UI fetches renames eagerly on connect/token change → renames passed in the `apply-variables` message → plugin renames variables before the main sync loop (only TokenManager-managed variables, checked via `tokenPath` plugin data, only if target name doesn't already exist).
- Files changed:
  - `packages/server/src/services/operation-log.ts` ~L59 (added `pathRenames?` to `OperationEntry`, `getPathRenames()` method)
  - `packages/server/src/routes/tokens.ts` ~L172 (group-rename), ~L772 (token-rename), ~L519 (batch-rename) — populate `pathRenames`
  - `packages/server/src/routes/operations.ts` ~L19 (new `GET /api/operations/path-renames` endpoint)
  - `packages/figma-plugin/src/shared/types.ts` ~L403 (added `renames?` to `ApplyVariablesMessage`)
  - `packages/figma-plugin/src/plugin/variableSync.ts` ~L6 (signature), ~L55 (rename pre-processing loop)
  - `packages/figma-plugin/src/plugin/controller.ts` ~L290 (pass `msg.renames`)
  - `packages/figma-plugin/src/ui/components/PublishPanel.tsx` ~L197 (renamesRef + fetch effect), ~L224 (include in payload)
- **Learnings for future iterations:**
  - Token rename→Figma variable rename requires knowing the old→new path pair; there's no way to infer it from the current Figma state alone. The operation log is the right source of truth.
  - The rename pre-pass must guard against: (a) non-TokenManager variables (check `tokenPath` plugin data), (b) target name already existing in the same collection, (c) same-name renames (no-op).
  - After renaming `oldVar.name = newFigmaName` in the pre-pass, `localVariables` already reflects the new name (JS objects are by reference), so `findVariableInList` in the main loop correctly finds the newly-renamed variable and updates its value without creating a duplicate.
  - Snapshot must be taken before renaming in the pre-pass (same snapshot map used for error rollback).

## 2026-04-05 - Fix parseInt || fallback treating 0 as falsy in paginated endpoints
- Replaced `parseInt(...) || default` with `isNaN(parsed) ? default : parsed` in all three paginated route handlers.
- Before: `parseInt('0', 10) || 200` → `200` (bug: client sending `?limit=0` or `?offset=0` got the default instead of 0)
- After: `isNaN(parsed) ? 200 : parsed` → `0` (correct; NaN from non-numeric input still falls back to the default)
- Files changed: `packages/server/src/routes/operations.ts` ~L10-11, ~L41-42; `packages/server/src/routes/tokens.ts` ~L96-97
- **Learnings for future iterations:**
  - `parseInt(str, 10) || fallback` is incorrect for numeric query params because `0` is a valid value but falsy — always use `isNaN` to distinguish "parse failed" from "value is zero"
  - The tokens.ts variant had an additional outer `limit ? ... : 200` guard (truthy check on the raw string) which was also wrong for `?limit=0` since `'0'` is a truthy string — the fix simplifies it to a single `isNaN` check

## 2026-04-05 - generator-service.ts silently swallows token resolution and set-fetch errors
- Removed 6 dead `.catch` handlers that silently swallowed errors in generator-service.ts:
  - L652: Removed `.catch(() => null)` on `resolveToken()` for `lastRunSourceValue` tracking; fixed the `!== null` check (which was logically wrong since `resolveToken` returns `undefined` for not-found, meaning `undefined !== null` would have accessed `.undefined.$value` causing a TypeError) to `if (resolved)`
  - L702/L773: Removed `.catch(() => ({}))` on `getFlatTokensForSet()` pre-run snapshot captures (single-brand and multi-brand paths). The function is pure in-memory and cannot throw; the catch created false expectations and would have resulted in an empty snapshot, meaning rollback after failure would delete all tokens in the set
  - L732/L822: Removed `.catch(() => ({}))` on `getFlatTokensForSet()` in rollback paths (same reasoning — dead code)
  - L965: Removed try-catch block around `resolveToken()` in `resolveConfigTokenRefs()` since `resolveToken` handles errors internally and always returns `undefined` for both not-found and resolution errors; replaced with direct `if (resolved)` check with an explanatory comment
- Files changed: `packages/server/src/services/generator-service.ts` ~L652, ~L702, ~L732, ~L773, ~L822, ~L965
- **Learnings for future iterations:**
  - `TokenStore.resolveToken()` and `getFlatTokensForSet()` both handle errors internally and never throw — any `.catch()` on them in calling code is dead code that can mask real errors in future
  - The `!== null` pattern for checking "token not found" is dangerous when the function returns `undefined` (not `null`) for not-found — always use a truthy check or `!== undefined` when the contract returns `undefined`
  - Pre-run snapshot captures should never silently fall back to `{}` — an empty snapshot means all pre-existing tokens would be deleted on rollback, which is worse than the original error
