## 2026-03-26 - Full undo/redo stack
- Replaced single-slot `useUndo` with a history stack (up to 20 entries) supporting both undo and redo
- `UndoSlot` gains optional `redo?: () => Promise<void>` — existing callers unchanged, redo opt-in
- `past[]` tracks undoable operations (newest at end), `future[]` tracks redoable operations after undo
- `pushUndo` appends to past and clears future (as in standard undo models)
- `executeUndo` pops from past, calls `slot.restore()`, pushes to future only if `slot.redo` exists
- `executeRedo` pops from future, calls `slot.redo()`, pushes back to past
- Keyboard: Cmd+Z = undo, Cmd+Shift+Z or Cmd+Y = redo
- No auto-dismiss (removed 8-second timer); toast persists until dismissed or stack exhausted
- `UndoToast` updated with optional Redo button and undo count badge (`[3] Delete token`)
- `App.tsx` updated to destructure and pass new hook return values
- Files changed: `packages/figma-plugin/src/ui/hooks/useUndo.ts` (full rewrite), `packages/figma-plugin/src/ui/components/UndoToast.tsx`, `packages/figma-plugin/src/ui/App.tsx` ~L134, ~L1292
- **Learnings for future iterations:**
  - Functional `setState(prev => ...)` is essential when multiple state updates within one handler need to see the latest state
  - `redo` is only pushed to future when `slot.redo` is defined — this lets existing single-action callers coexist without redo
  - The `executingRef` guard prevents double-invocation if a key is held down

## 2026-03-26 - Flow audit & overhaul: Inline Quick-Edit
- Fixed Escape key bug: pressing Escape in an inline text editor was silently saving the value instead of cancelling. Root cause: `setInlineEditActive(false)` is an async state update; when the input disappears from DOM React fires `onBlur`, which called `handleInlineSubmit()` with the stale `inlineEditActive=true` closure value.
- Fix: added `inlineEditEscapedRef = useRef(false)`, set it synchronously in the Escape handler, and check+reset it at the top of the `onBlur` handler before calling `handleInlineSubmit`.
- Files changed: `packages/figma-plugin/src/ui/components/TokenList.tsx` ~L1956, ~L2492-2499
- **Learnings for future iterations:**
  - When pressing Escape to cancel an edit causes DOM removal, the resulting `onBlur` fires with the old memoized closure — use a `ref` (not state) to synchronously flag the cancellation before the blur fires. This is the same pattern as `renameGroupEscapedRef` already in the same component.
  - Enter key is safe because React re-renders before blur fires when the input disappears, so the second `handleInlineSubmit` call sees `inlineEditActive=false` and exits early.

## 2026-03-26 - Flow audit & overhaul: Token Binding
- Audited SelectionInspector.tsx — friction points: action buttons were hidden behind hover (opacity-0), no way to swap an existing binding without remove+rebind, no keyboard navigation in the bind token search, no resolved value preview for dimension/number tokens in search results
- Changed action button opacity from `opacity-0` to `opacity-40` — buttons are now always slightly visible, giving a persistent affordance that properties are bindable
- Added "Change binding" button (pencil icon) for already-bound properties — opens the bind panel directly without requiring remove + rebind
- Added `bindSelectedIndex` state + ↑/↓/Enter keyboard navigation in the bind panel token list
- Moved `bindCandidates` computation out of the IIFE into the `.map()` callback so it's accessible in both the input `onKeyDown` and the list render
- Added resolved value display for `dimension` and `number` tokens in bind results (e.g. `16px`, `0.5`) alongside the type badge
- Updated bind panel header to show "Change binding for X" vs "Bind X" depending on whether the property is already bound
- Files changed: `packages/figma-plugin/src/ui/components/SelectionInspector.tsx` ~L156, ~L218, ~L225, ~L471, ~L540, ~L575
- **Learnings for future iterations:**
  - `opacity-40 group-hover:opacity-100` is the right affordance pattern for actions that should be discoverable but not visually dominant — fully invisible (opacity-0) means users never learn they exist
  - When an item can be in a "bound" state, you need a "change" action alongside "remove" — the remove+rebind round-trip is too many steps
  - Keyboard nav in search lists requires candidates to be computed before the input's `onKeyDown`, not inside a deferred IIFE — move the candidates computation to the enclosing scope
  - Arrow key handlers must call `e.preventDefault()` to suppress scroll on the parent container
