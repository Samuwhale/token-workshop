## 2026-04-05 - BatchEditor lacks sub-property targeting for composite tokens
- Added a "Sub-property" selector row to BatchEditor that appears when composite tokens (shadow, typography, border, transition) are in the selection. The dropdown lists all targetable sub-properties grouped by composite type (e.g. "shadow: color", "shadow: blur", "typography: fontSize"). When a sub-property is selected:
  - Color sub-props (shadow.color, border.color): the existing opacity and color adjust controls (lighten/darken/saturate/desaturate/hue) are activated/shown and apply to that sub-property on all matching tokens.
  - Numeric sub-props (shadow.blur, typography.fontSize, border.width, etc.): the existing numeric transform control (×/÷/+/−) is activated/shown and applies to that sub-property.
  - Preview rows render color swatches (for color sub-props) or before→after values (for numeric sub-props), one per token.
  - Array-valued shadows (multiple shadow instances) are handled by transforming each instance's sub-property.
- `hasColorTarget` and `hasNumericTarget` derived consts unify regular token targets with composite sub-prop targets so `opacityActive`, `colorAdjustActive`, and `numericTransformActive` gate on both.
- Skip counters (`skippedNotColor`/`skippedNotNumeric`) were updated to not double-count composite tokens when a sub-prop is selected — they're handled by the dedicated composite apply loop.
- Files changed: `packages/figma-plugin/src/ui/components/BatchEditor.tsx` ~L131 (COMPOSITE_SUB_PROPS_BY_TYPE), ~L207 (compositeSubPropKey state), ~L358 (composite memos block), ~L419 (hasColorTarget/hasNumericTarget), ~L567 (composite apply loop in handleApply), ~L700 (cleanup reset), ~L1148 (composite sub-prop UI section), visibility updates at ~L1118, ~L1136, ~L1163 (hasColors/hasScalable → hasColorTarget/hasNumericTarget)
- **Learnings for future iterations:**
  - Declaration order matters for TDZ: composite memos must be placed BEFORE `hasColorTarget`/`hasNumericTarget` (which reference `compositeSubPropKind`/`compositeSubPropTargets`). The safest rule: any derived const that feeds another derived const must appear first in the function body.
  - `hasColorTarget = hasColors || (compositeSubPropKind === 'color' && compositeSubPropTargets.length > 0)` is the correct pattern for extending an existing "active" check to include composite sub-prop targets — keep a single `opacityActive`/`colorAdjustActive` that covers both.

## 2026-04-05 - Plugin sandbox variableSync rollback is all-or-nothing for the deletion phase
- Fixed per-variable granularity in both rollback paths (`applyVariables` catch block and `revertVariables`). Previously, if any single restore operation failed (e.g., the `scopes` setter on one variable), the entire deletion phase was skipped for ALL created variables and collections, leaving orphan variables in the document. Now each restore failure is tracked by its specific varId, and the deletion phase always runs — only variables that individually failed their restore are skipped (defensively; in practice `createdVariableIds` and `variableSnapshots` are always disjoint sets, so all created orphans get cleaned up).
- In `applyVariables`: extracted `varSnapshotEntries` array before building `restoreTasks` so indices correlate reliably with `restoreResults`; replaced the all-or-nothing `if/else` with per-varId tracking via `failedRestoreVarIds` Set.
- In `revertVariables`: added `failedRestoreVarIds` Set; each property-setter catch block sets `varFailed = true`; added `failedRestoreVarIds.add(varId)` at end of each task; removed the skip-all `else` guard.
- Files changed: `packages/figma-plugin/src/plugin/variableSync.ts` ~L168 (applyVariables rollback), ~L249 (revertVariables)
- **Learnings for future iterations:**
  - `variableSnapshots` (existing vars) and `createdVariableIds` (new vars) are always disjoint — a restore failure for an existing variable should never block deletion of a newly-created orphan variable. When these two concerns are conflated in an if/else, any property-setter failure (even `scopes`) blocks the entire cleanup path.
  - To correlate `Promise.allSettled` results with their input varIds, build tasks from a named array (`varSnapshotEntries`) rather than calling `Array.from(map.entries())` twice — even though Map iteration order is deterministic, using the same array reference makes the index correlation explicit and safe.
  - In `revertVariables`, failures accumulate inline via `failures.push()` rather than being thrown, so tracking per-variable failure requires a `varFailed` local flag per async task rather than checking `r.status === 'rejected'`.

## 2026-04-05 - Publish readiness checks go stale silently after token edits
- Root cause: when tokens are edited *during* an active check run, the auto-recheck effect (triggered by tokenChangeKey change) fires but is blocked by `isRunningRef.current = true`. After the check completes it sets `checksRunAtKey` to the old key value, leaving `isReadinessOutdated = true` with no further trigger to re-run — the tokenChangeKey effect won't fire again because the key hasn't changed.
- Fix: added `latestTokenChangeKeyRef` (synced to `tokenChangeKey` via useEffect) in `useReadinessChecks.ts`. In the `finally` block of `runReadinessChecks`, after clearing `isRunningRef`, compare `thisRunKey` (from the closure) against `latestTokenChangeKeyRef.current`. If they differ (edits accumulated in-flight), schedule a follow-up via `Promise.resolve().then(() => runReadinessChecksRef.current())`.
- Files changed: `packages/figma-plugin/src/ui/hooks/useReadinessChecks.ts` ~L67 (latestTokenChangeKeyRef + useEffect), ~L154 (finally follow-up logic)
- **Learnings for future iterations:**
  - `isRunningRef` mutex is correct but creates a silent gap: the last auto-recheck attempt that was blocked never gets retried. Any ref-guarded async operation should check for accumulated work on completion.
  - `tokenChangeKey` in the `useCallback` closure is the value captured when that callback version was created, not the live value — that's why a separate `latestTokenChangeKeyRef` is needed to detect in-flight drift.

## 2026-04-05 - Inline popover edits bypass undo stack
- Fixed `handleInlineSave` and `handleDescriptionSave` in `useTokenSave.ts` to use `perSetFlat?.[setName]?.[path]` as the primary source for capturing old values before a save, falling back to `allTokensFlat[path]`. This fixes two related bugs:
  1. **Undo skipped for theme-disabled tokens**: When a token lives in a set that's "disabled" by the active theme, `allTokensFlat` (= `themedAllTokensFlat`) doesn't contain it. The `oldEntry` would be `undefined`, silently skipping `pushUndo`.
  2. **Undo restores resolved values**: `allTokensFlat` has alias-resolved values. For composite tokens (shadow, typography, etc.) with aliased sub-properties (e.g. `color: "{color.primary}"`), undo would restore the resolved hex value (`#FF0000`) instead of the alias reference, permanently destroying the alias.
  Using `perSetFlat[setName][path]` captures the raw, unresolved per-set value, so undo correctly restores the original file content.
- Also updated the `useCallback` dependency arrays for both handlers to include `perSetFlat`.
- Files changed: `packages/figma-plugin/src/ui/hooks/useTokenSave.ts` ~L37, ~L45, ~L90, ~L95, ~L140
- **Learnings for future iterations:**
  - `allTokensFlat` in TokenList's context is `themedAllTokensFlat` (cross-set, alias-resolved). It is NOT the per-set raw value map. Always use `perSetFlat[setName][path]` when you need the raw pre-save value for undo of an inline edit.
  - `perSetFlat` is built from `fetchAllTokensFlatWithSets` without `resolveAllAliases`, so its values preserve alias references intact. `allTokensFlat` runs `resolveAllAliases` on top.

## 2026-04-05 - No undo operation grouping
