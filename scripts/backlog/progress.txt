## 2026-03-27 - No conflict detection for concurrent edits ‚Äî if a token is modified on the server while the editor is open, saving silently overwrites the server version
- Added pre-save conflict detection to TokenEditor: when saving (PATCH), the editor re-fetches the current server state and compares it (via `JSON.stringify`) to the snapshot captured on initial load. If they differ, a ConfirmModal is shown: "Token modified on server ‚Äî overwrite?" If the user confirms, `handleSave(true)` is called with `forceOverwrite=true` to skip the check. If the conflict check fetch fails (network error), save proceeds normally to avoid blocking the user.
- Files changed: `packages/figma-plugin/src/ui/components/TokenEditor.tsx` ~L171 (new state/ref), ~L218 (snapshot capture), ~L397 (conflict check in handleSave), ~L1332 (conflict ConfirmModal in JSX)
- **Learnings for future iterations:**
  - Snapshot comparison uses `JSON.stringify` on the raw server response token object ‚Äî both the initial load and the conflict check hit the same endpoint so key ordering is consistent.
  - `forceOverwrite` parameter on `handleSave` is the simplest way to let the conflict modal re-invoke save without re-triggering the check ‚Äî no additional state needed.
  - Network failures during the conflict check are silently swallowed to avoid false-positive conflict blocks.

## 2026-03-27 - Identical fetch-and-error pattern duplicated across 10+ call sites ‚Äî extract a shared `apiFetch` utility
- Created `packages/figma-plugin/src/ui/shared/apiFetch.ts` exporting `ApiError` (extends Error, adds `status`) and `apiFetch<T>(url, options?)` which throws `ApiError` on non-OK responses after extracting `{ error }` from the response body.
- Updated 10 call sites across 4 files: `useGeneratorDialog.ts` (preview fetch, save fetch), `App.tsx` (rename set, create set, delete set), `ExportPanel.tsx` (export endpoint, batch save), `ImportPanel.tsx` (fetchSets, Figma Variables import batch, JSON/styles import batch). Each site had its own `res.ok` check + `res.json().catch(() => ({}))` error extraction ‚Äî now replaced with a single `await apiFetch(...)` call.
- The 409-conflict case in set creation was preserved using `catch (err) { if (!(err instanceof ApiError && err.status === 409)) throw ... }`.
- Files changed: `packages/figma-plugin/src/ui/shared/apiFetch.ts` (new), `packages/figma-plugin/src/ui/hooks/useGeneratorDialog.ts` ~L1,L211,L320, `packages/figma-plugin/src/ui/App.tsx` ~L40,L565,L657,L673, `packages/figma-plugin/src/ui/components/ExportPanel.tsx` ~L1,L217,L420, `packages/figma-plugin/src/ui/components/ImportPanel.tsx` ~L1,L107,L355,L463
- **Learnings for future iterations:**
  - `ApiError` should extend `Error` and carry `status: number` ‚Äî this lets callers distinguish "network down" (TypeError) from "server error" (ApiError) and allows status-based logic (e.g. 409 = already exists = OK).
  - When a call site has `if (!res.ok) { ... return; }` and a separate `res.json()` on success, `apiFetch` simplifies it to a single awaited call whose return value is the parsed JSON.
