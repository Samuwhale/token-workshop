## 2026-04-04 - TokenStore rebuildFlatTokens() non-atomic + missing rebuild on error path
- Fixed `rebuildFlatTokens()` to build into a new `Map` and assign atomically (`this.flatTokens = newMap`), eliminating the window where concurrent SSE/resolveTokens readers could observe an empty or partial token map
- Added `this.rebuildFlatTokens()` in the `updateToken` catch block after `restoreSnapshots()`: `updateToken` mutates token properties in-place (`existing.$value = …`), which corrupts the existing `flatTokens` entries; `restoreSnapshots` replaces the tree with a deep clone but `flatTokens` still held references to the mutated objects — the rebuild re-derives from the restored snapshot
- `setToken`, `deleteToken`, `createGroup`, `updateGroupMeta` don't need error-path rebuilds because they either replace whole objects (not mutate in-place) or run inside `withBatch` (whose `endBatch` always calls `rebuildFlatTokens`)
- Files changed: `packages/server/src/services/token-store.ts` ~L266, ~L800
- **Learnings for future iterations:**
  - In-place mutation of objects already referenced by a derived data structure (flatTokens) is the subtle bug — `setTokenAtPath` replaces the node so old references stay valid, but `updateToken`'s property-assignment pattern corrupts the referenced object
  - Atomic Map swap is a one-liner: build into a local `newMap`, assign to `this.flatTokens` at the end; readers never see a partial state

## 2026-04-04 - Server-side validation gaps allow invalid state to persist until runtime failure
- Fixed three server-side validation gaps:
  1. `operation-log.ts` TOCTOU race in `rollback()`: wrapped the entire rollback body inside `withLock()` (a single lock for the whole operation). Previously the `rolledBack` check and the final `withLock` mark-and-persist were separate, allowing concurrent requests to both pass the check before either set `rolledBack = true`. Removed the inner nested `withLock()` call since we now hold the lock for the whole operation and `pushAndPersist` is documented to be called while holding the lock.
  2. `generator-service.ts` circular dependency check: `create()` and `update()` now call `buildDependencyOrder()` immediately after updating `this.generators`. On cycle detection, the change is reverted in-memory and a `BadRequestError` (HTTP 400) is thrown before `saveGenerators()` is called. Previously cycles were only discovered at runtime when a generator fired.
  3. `git-sync.ts` partial conflict resolution: `resolveAllConflicts()` now checks that every conflict region (0..regionCount-1) has a corresponding choice entry. Missing indices are collected and reported in a `BadRequestError` before any file writes occur, preventing merge markers from persisting in staged files.
- Files changed:
  - `packages/server/src/services/operation-log.ts` ~L332-413 (rollback wrapping)
  - `packages/server/src/services/generator-service.ts` ~L133-162 (create/update cycle check)
  - `packages/server/src/services/git-sync.ts` ~L384-399 (complete-resolution check)
- **Learnings for future iterations:**
  - When a mutex (`withLock`) is used, the check and the guarded mutation must both happen inside the same lock scope. Checking a condition before acquiring the lock and setting it after is a TOCTOU race even in single-threaded async code, because `await` yields between the check and the set.
  - For in-memory state mutations with async validation, the pattern is: apply mutation → validate → revert on failure → save. This makes the validation path clean with no intermediate writes.
  - Partial-resolution validation (checking all indices are covered) belongs alongside range validation in the same step, before any file I/O.
