## 2026-04-05 - Generator dependency graph visualization
- Added cross-generator dependency flow visualization to the NodeGraph view. When generator B's `sourceToken` falls within generator A's `targetGroup.*` output space, a dashed accent-colored bezier edge now appears from A's output node curving down-right-left to B's source node. Generators are also topologically sorted in the layout (producers above consumers) so the visual flow reads top-to-bottom.
- Implementation: added `DependencyEdge` interface, `computeDependencyEdges()` (detects upstream→downstream relationships by prefix matching `sourceToken.startsWith(targetGroup + '.')`), and `topoSortGenerators()` (Kahn's BFS topological sort to order generators by dependency) to `nodeGraphTypes.ts`. `generatorsToGraph` now sorts via `topoSortGenerators` before assigning row positions. `NodeGraphCanvas.tsx` imports `computeDependencyEdges`, calls it with `useMemo`, and renders the edges using a custom `depEdgePath()` function that routes right of the output column then curves down-left to the source node. Edge includes a glow/hit layer, a dashed stroke, an arrowhead circle at the destination, and a monospace token-path label at the midpoint.
- Files changed: `packages/figma-plugin/src/ui/components/nodeGraph/nodeGraphTypes.ts` ~L83 (DependencyEdge, computeDependencyEdges, topoSortGenerators, generatorsToGraph update), `packages/figma-plugin/src/ui/components/nodeGraph/NodeGraphCanvas.tsx` ~L1 (import), ~L15 (depEdgePath), ~L170 (useMemo), ~L515 (render)
- **Learnings for future iterations:**
  - `NodeGraphState` did not need to be changed — dependency edges are fully derivable from the generators array, so computing them in `useMemo` in the canvas keeps the mutable graph state clean.
  - The standard `edgePath()` bezier (horizontal control points) produces weird S-curves when x1 > x2 (right-to-left). The `depEdgePath()` function uses `rightX = max(x1,x2) + 60` as the routing waypoint, producing a "paperclip" arc that routes around the right column and arrives at the destination with a left-to-right tangent.
  - Topological sort using Kahn's algorithm (BFS on in-degree) is the right primitive for DAG-ordering generators; cycle members fall back to original order safely.

## 2026-04-05 - Git-sync applyDiffChoices continues after partial checkout failures
- The server-side `applyDiffChoices` already correctly tracks `pullFailedFiles`, only commits successfully-checked-out files, and returns failure details in `ApplyDiffResult`. The bug was entirely in the UI: `useGitDiff.ts:applyDiff` called `apiFetch` but ignored the returned data, treating every HTTP 200 as full success even when `pullFailedFiles.length > 0`. Fixed by: (1) adding `ApplyDiffResponse` interface mirroring `ApplyDiffResult`; (2) reading the response and building descriptive error messages for pull failures, commit failures, and push failures; (3) when any failures exist, showing the error via `setGitError` and calling `computeDiff()` to refresh the diff view to show remaining work; (4) only closing the diff view on full success.
- Files changed: `packages/figma-plugin/src/ui/hooks/useGitDiff.ts` ~L33 (ApplyDiffResponse interface), ~L110 (applyDiff callback)
- **Learnings for future iterations:**
  - The server side of `applyDiffChoices` was already correct — it committed only successful checkouts and surfaced `pullFailedFiles`. The gap was purely UI-side.
  - `apiFetch` only throws on HTTP 4xx/5xx; a partial-success 200 response with failure details in the body is silently treated as success unless the caller inspects the returned data.
  - After partial apply, calling `computeDiff()` (not just `fetchStatus()`) is the right refresh — it updates the diff view so the user sees which files remain outstanding.

## 2026-04-05 - Resolver-store delete catches all unlink errors silently
- Fixed `resolver-store.ts` `delete()` to only swallow `ENOENT` (file already gone) and re-throw all other errors (permission-denied, disk-full, etc.). Previously the `finally` block always cleared the write guard and the code always removed the resolver from memory regardless of unlink outcome. Now non-ENOENT errors propagate to the caller before the in-memory state is mutated, preserving disk/memory consistency.
- Files changed: `packages/server/src/services/resolver-store.ts` ~L156
- **Learnings for future iterations:**
  - When silencing file-deletion errors, always be explicit: catch the error variable, check `.code === 'ENOENT'`, and re-throw anything else. A bare `catch {}` or `catch { // may already be gone }` is almost always wrong for anything beyond trivial scripts.
  - If a `finally` block unconditionally mutates state (here: `resolvers.delete(name)`), moving the mutation after the try/catch body and handling it explicitly in the ENOENT branch makes the control flow much clearer.

## 2026-04-05 - Merge conflict resolver in GitSubPanel lacks diff highlighting
- Added LCS-based line-level diff highlighting to the conflict resolver in `GitSubPanel.tsx`. Each conflict region now computes a diff between `ours` and `theirs` using a standard LCS DP algorithm (capped at 300 lines per side to prevent quadratic blowup). Changed lines are highlighted: green tint for your-version changes, accent/blue tint for server-version changes. Unchanged lines remain neutral. Also added a summary banner per region showing "N lines changed in yours · M in server's". A new `DiffPre` component renders diff-annotated `<pre>` content line-by-line. The `useMemo` on `regionDiffs` ensures diffs are computed once per conflict set, not on every render.
- Files changed: `packages/figma-plugin/src/ui/components/publish/GitSubPanel.tsx` ~L1 (imports), ~L8 (DiffLine interface), ~L12 (computeLineDiff), ~L58 (DiffPre), ~L86 (regionDiffs memo), ~L285 (conflict region rendering)
- **Learnings for future iterations:**
  - LCS-based line diff is the right primitive for text regions in merge conflicts — it correctly identifies unchanged anchor lines and marks diverging lines as changed, without needing to split by words or characters.
  - For two-column side-by-side diffs without alignment, computing independent changed-line sets per side is sufficient and much simpler than aligned diffs. Alignment would require inserting blank spacer rows which adds complexity for a narrow plugin panel.
  - The `useMemo` pattern (computing all diffs in a single pass over `git.mergeConflicts`) keeps the rendering path clean — no computation in JSX.

## 2026-04-05 - Plugin sandbox variableSync rollback is all-or-nothing for the deletion phase
- Fixed per-variable granularity in both rollback paths (`applyVariables` catch block and `revertVariables`). Previously, if any single restore operation failed (e.g., the `scopes` setter on one variable), the entire deletion phase was skipped for ALL created variables and collections, leaving orphan variables in the document. Now each restore failure is tracked by its specific varId, and the deletion phase always runs — only variables that individually failed their restore are skipped (defensively; in practice `createdVariableIds` and `variableSnapshots` are always disjoint sets, so all created orphans get cleaned up).
- In `applyVariables`: extracted `varSnapshotEntries` array before building `restoreTasks` so indices correlate reliably with `restoreResults`; replaced the all-or-nothing `if/else` with per-varId tracking via `failedRestoreVarIds` Set.
- In `revertVariables`: added `failedRestoreVarIds` Set; each property-setter catch block sets `varFailed = true`; added `failedRestoreVarIds.add(varId)` at end of each task; removed the skip-all `else` guard.
- Files changed: `packages/figma-plugin/src/plugin/variableSync.ts` ~L168 (applyVariables rollback), ~L249 (revertVariables)
- **Learnings for future iterations:**
  - `variableSnapshots` (existing vars) and `createdVariableIds` (new vars) are always disjoint — a restore failure for an existing variable should never block deletion of a newly-created orphan variable. When these two concerns are conflated in an if/else, any property-setter failure (even `scopes`) blocks the entire cleanup path.
  - To correlate `Promise.allSettled` results with their input varIds, build tasks from a named array (`varSnapshotEntries`) rather than calling `Array.from(map.entries())` twice — even though Map iteration order is deterministic, using the same array reference makes the index correlation explicit and safe.
