## 2026-03-26 - Race condition: concurrent file watcher rebuilds
- Added `scheduleRebuild(event)` private method to `TokenStore` that debounces `rebuildFlatTokens()` by 50ms and batches pending `ChangeEvent`s, flushing them after the rebuild completes. Replaced direct `rebuildFlatTokens()` + `emit()` calls in the three watcher handlers (`change`, `add`, `unlink`) with `scheduleRebuild()`. Also clears the timer in `shutdown()` to prevent leaks.
- Files changed: `packages/server/src/services/token-store.ts` ~L20, L67
- **Learnings for future iterations:**
  - Only debounce the watcher-triggered rebuilds — CRUD methods (createToken, updateToken, etc.) still call `rebuildFlatTokens()` directly since callers expect immediate state consistency.
  - The debounce timer must be cleared in `shutdown()` to avoid timers firing after the watcher is closed.
  - Coalescing events into `_pendingWatcherEvents` ensures all SSE listeners still receive one event per changed file even when the rebuild is batched.

- Implemented full multi-dimensional theme model: `ThemeDimension` (e.g. "Color Mode") contains mutually exclusive `ThemeOption`s (e.g. "Light"/"Dark"). Multiple dimensions can be active simultaneously, creating a matrix of combinations.
- **Data model**: Replaced flat `Theme[]` with `ThemeDimension[]` in `types.ts` — new types: `ThemeDimension { id, name, options }`, `ThemeOption { name, sets }`, `ActiveThemes = Record<string, string>` (dimId → optionName)
- **Server API**: Complete rewrite of `routes/themes.ts` — new endpoints: `GET /api/themes` → `{ dimensions }`, `POST /api/themes/dimensions`, `PUT /api/themes/dimensions/:id`, `DELETE /api/themes/dimensions/:id`, `POST /api/themes/dimensions/:id/options`, `DELETE /api/themes/dimensions/:id/options/:optionName`
- **Token store**: Updated `renameSet` in `token-store.ts` to traverse `dimension.options[]` instead of flat `theme[]`
- **Controller**: Changed `get-active-theme`/`set-active-theme` to `get-active-themes`/`set-active-themes` using a map object instead of single string
- **App.tsx**: Multi-dimensional switcher bar (one dropdown per dimension), multi-option token resolution that merges all active options in dimension order
- **ThemeManager.tsx**: Complete rewrite — dimensions as top-level cards, options inside each card with the same Off/Base/On set matrix, inline option creation per dimension, coverage gaps per option
- **ThemeCompare.tsx**: Updated to accept `dimensions` prop and build a flat option list for comparison
- Files changed: `packages/core/src/types.ts` ~L142-161, `packages/core/src/index.ts` ~L43-47, `packages/server/src/routes/themes.ts` (full rewrite), `packages/server/src/services/token-store.ts` ~L261-272, `packages/figma-plugin/src/plugin/controller.ts` ~L68-82, `packages/figma-plugin/src/ui/App.tsx` ~L239-357+L1692-1748+L2124+L2142, `packages/figma-plugin/src/ui/components/ThemeManager.tsx` (full rewrite), `packages/figma-plugin/src/ui/components/ThemeCompare.tsx` ~L6-80
- **Learnings for future iterations:**
  - The server tsconfig uses `"paths": { "@tokenmanager/core": ["../core/src"] }` with composite project references — server imports directly from core source, not dist. Run core build if server fails with "no exported member" errors.
  - ActiveThemes stored as `Record<string, string>` (dimensionId → optionName) in both localStorage and Figma clientStorage. Key prefix changed from `active-theme:` to `active-themes:`.
  - Token resolution with multiple dimensions: iterate dimensions in order, for each apply its active option's source sets then enabled sets. Later dimensions override earlier ones for same paths.

## 2026-03-26 - Import from Figma Variables
- Updated `readFigmaVariables()` in controller.ts to return ALL modes per collection instead of only the first mode. Now sends `{ type: 'variables-read', collections: [{name, modes: [{modeId, modeName, tokens:[...]}]}] }`.
- Rewrote ImportPanel.tsx variables flow: instead of a flat token list going to one set, shows a collection/mode → set-name mapping UI. Each collection×mode pair shows token count and an editable set name (default: `{collection}/{mode}` when multi-mode, `{collection}` when single-mode). Users can enable/disable individual modes and rename the target sets before importing. Styles import flow is unchanged.
- Files changed: `packages/figma-plugin/src/plugin/controller.ts` ~L460, `packages/figma-plugin/src/ui/components/ImportPanel.tsx` (full rewrite)
- **Learnings for future iterations:**
  - The prior implementation silently dropped all modes except the first — the fix is non-trivial because you need the outer collection loop to drive per-mode token extraction
  - `modeKey(collectionName, modeId)` pattern (pipe-separated) is a safe composite key since collection names and modeIds won't contain pipes
  - For variables import, coarse-grained selection (per mode, not per token) is the right UX since there can be hundreds of tokens across many modes

## 2026-03-26 - Bulk remap across selection
- Implemented a "Remap Bindings" panel in the Inspect tab. A "Remap" button in the sync controls bar opens a collapsible panel with old→new path rows, a scope toggle (Selection/Page), and an Execute button. - Files changed: `packages/figma-plugin/src/shared/types.ts` ~L157, `packages/figma-plugin/src/plugin/controller.ts` ~L48 and ~L1095, `packages/figma-plugin/src/ui/components/SelectionInspector.tsx` ~L158, ~L182, ~L296, ~L508
