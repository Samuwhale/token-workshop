## 2026-04-03 - git-sync.ts operations have no mutual exclusion
- Added a `withLock<T>(fn)` promise-chain mutex to the `GitSync` class. All git-mutating operations now serialize behind a single `lockChain` promise so concurrent calls can never interleave git add/checkout/commit commands.
- Methods wrapped: `init`, `commit`, `push`, `pull`, `resolveFileConflict`, `abortMerge`, `finalizeMerge`, `checkout`, `createBranch`, `setRemote`, `applyDiffChoices`. Read-only methods (`status`, `log`, `fetch`, `getConflictedFiles`, etc.) are left unlocked.
- The lock pattern: `next = lockChain.then(() => fn()); lockChain = next.then(() => {}, () => {}); return next;` — errors propagate to callers but never break the chain for subsequent callers.
- Files changed: `packages/server/src/services/git-sync.ts` ~L199
- **Learnings for future iterations:**
  - The same promise-chain mutex pattern from TokenStore (`withLock`) works identically here — no external deps needed.
  - Only operations that mutate the git index (staging area), working tree, or history need the lock. Pure-read operations (diff, log, show) are safe to run concurrently.
  - `this.lockChain = next.then(() => {}, () => {})` (not `.catch(() => {})`) ensures the chain stays in the fulfilled state regardless of outcome, preventing subsequent callers from being blocked by a rejected promise.

## 2026-04-03 - TokenStore write guards leak on delete and clearAll
- Added `try/finally` blocks to `deleteSet()` and `clearAll()` in `token-store.ts` to call `_clearWriteGuard()` after each `fs.unlink` completes (success or throw), rather than relying on the chokidar 'unlink' watcher event to clear the guard. This eliminates the window where a file restored externally (e.g., `git checkout`) could be silently suppressed.
- Refactored `ResolverStore.delete()` in `resolver-store.ts` to move `_startWriteGuard` outside the try/catch and add a `finally` block — guard is now cleared on both success and error paths (previously only cleared on the error path).
- Files changed: `packages/server/src/services/token-store.ts` ~L623, ~L648; `packages/server/src/services/resolver-store.ts` ~L155
- **Learnings for future iterations:**
  - For write operations (rename/atomic overwrite), the watcher 'change'/'add' event clears the guard. For delete (`fs.unlink`), the 'unlink' event fires — but if an external tool restores the file between unlink and the watcher event, the guard may suppress the 'add' event for the restored file.
  - The `renameSet` `try/finally` pattern is the correct one for all operations: start the guard before the fs operation, clear it in `finally` after completion. Don't rely on watcher callbacks to be the sole guard-clearing mechanism.
  - The watcher 'unlink' handler calling `sets.delete(name)` after we've already deleted it is idempotent (Map.delete on missing key is a no-op); the extra `scheduleRebuild` emission is acceptable noise.

- Added `tokenChangeKey?: number` prop to `PublishPanelProps` and passed it from App.tsx.
- Added `checksRunAtKey` (number | null) and `checksStale` (boolean) state to track when results are outdated.
- After `runReadinessChecks` succeeds: records `checksRunAtKey = tokenChangeKey`, clears `checksStale`, saves key to `localStorage` (`tm_readiness_change_key`).
- Derived `isReadinessOutdated`: true when checks exist AND (`checksStale` OR `tokenChangeKey !== checksRunAtKey`).
- Added "Outdated" badge (amber pill) in the readiness header when `isReadinessOutdated && !readinessLoading`; also turns the status dot amber.
- Suppressed the "Ready" green badge when outdated (it's no longer accurate).
- Set `checksStale = true` after each apply operation: apply-vars modal, apply-styles modal, apply-diff modal, and `runPublishAll`.
- Auto-run on tab activation: mount effect reads `tm_readiness_change_key` from localStorage; if `tokenChangeKey > storedKey`, auto-runs checks immediately.

- Files changed: `packages/figma-plugin/src/ui/components/PublishPanel.tsx` ~L28-100, ~L125-240, ~L305-395; `packages/figma-plugin/src/ui/App.tsx` ~L2513; `packages/figma-plugin/standalone/validate.mjs` ~L158
- **Learnings for future iterations:**
  - `waitUntil: 'networkidle'` in Playwright will time out whenever the plugin has ongoing server polling (health checks, etc.) — always use `waitUntil: 'load'` for plugin UI validation.
  - The "stable callback ref" pattern (`useRef` + assignment on every render, called from mount effect) is the clean way to call a `useCallback` function from a mount-only effect without listing all its deps.
  - Persisting the "last run at change key" in localStorage enables auto-run staleness detection across component remounts (PublishPanel unmounts/remounts each time you navigate away from the Publish tab).
