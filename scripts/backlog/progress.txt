## 2026-03-27 - `docs.ts` style attribute built with `escapeHtml` but not `escapeCssValue` — CSS injection possible via adversarial token values
- Applied `escapeCssValue()` to each individual CSS property value (fontFamily, fontSize, fontWeight, lineHeight, letterSpacing) when building the typography specimen style string, replacing the prior `escapeHtml(style)` on the concatenated string. `escapeHtml` escapes HTML metacharacters but leaves `;` intact, which allows injecting additional CSS properties (e.g. `fontFamily: "Arial; background: red"`). `escapeCssValue` strips `;`, `"`, `{`, `}`, `<`, `>` from each value individually, keeping the property delimiters in the constructed string valid.
- Files changed: `packages/server/src/routes/docs.ts` ~L65-70
- **Learnings for future iterations:**
  - `escapeCssValue` must be applied per-value, not to the full multi-property style string — applying it to the full string would strip the valid `;` delimiters between properties.
  - The pattern was already correctly used in `renderColorTokens` (line 44); the typography function just missed it.

## 2026-03-27 - bugfix-pass: wildcard route params slash→dot normalization
- Bug: The server's wildcard `*` route handlers for `/tokens/:set/*` and `/tokens/:set/dependents/*` passed `request.params['*']` directly to token store functions which split on `.`. But several client call sites encode token paths as slash-separated URLs (e.g., `colors/primary/accent` from `path.split('.').map(encodeURIComponent).join('/')`). This caused: (1) `handleInlineSave` PATCH calls to return 404 and silently fail — inline edits appeared to save but values reverted on refresh; (2) `handleConfirmPromote` PATCH calls to silently fail; (3) the `getDependents` check to always return 0 — the "N tokens reference this token" confirmation dialog was never shown before renames.
- Fix: Added `.split('/').join('.')` normalization to `request.params['*']` in all 5 wildcard route handlers (dependents GET, token GET/POST/PATCH/DELETE). Dot-separated paths (no `/`) are unaffected; slash-separated paths are converted to dot-separated before store lookup.
- Files changed: `packages/server/src/routes/tokens.ts` ~L173, ~L259, ~L283, ~L316, ~L343
- **Root cause:** Two URL-construction conventions coexisted in the client — some handlers used raw dot-separated paths, others (inline save, promote-to-alias) used per-segment `encodeURIComponent` joined with `/`. The server never normalized between them.

## 2026-03-27 - housekeeping: remove `catch (err: any)` anti-pattern in sets.ts
- Changed two catch blocks from `catch (err: any)` with `err?.message ?? String(err)` to `catch (err)` with `err instanceof Error ? err.message : String(err)`
- Files changed: `packages/server/src/routes/sets.ts` ~L67, ~L89
- **Why:** TypeScript's default catch type is `unknown` — annotating `err: any` opts out of type safety and allows accessing arbitrary properties without type guards. All other catch blocks in the same file already used the safe pattern.

## 2026-03-27 - bugfix-pass: SyncPanel concurrent read-variables requests clobbered by shared refs
- Bug: `computeVarDiff` and `runReadinessChecks` both wrote to the same `varCorrelationIdRef` and `varReadResolveRef`. If both ran concurrently (e.g., auto-run on mount while the user clicked "Run readiness checks"), the second call overwrote the cid/resolver, causing the first call's `variables-read` response to fail the `correlationId` check — the message was silently dropped and that call hung for the full 10-second timeout before rejecting.
- Fix: Replaced the two single-slot refs with a single `varPendingRef = useRef<Map<string, resolver>>(new Map())`. Each call registers its resolver by its unique cid; the message handler looks up by correlationId, calls the resolver, and deletes the entry. Concurrent calls no longer interfere.
- Files changed: `packages/figma-plugin/src/ui/components/SyncPanel.tsx` ~L70-71, ~L123-135, ~L192-195, ~L251-260
- **Root cause:** The original code assumed only one `read-variables` request could be in flight at a time — a valid assumption when the two callers were separate buttons, but violated by the auto-run `useEffect` which fires immediately on mount while the user might also trigger the readiness check.
