## 2026-03-27 - bugfix-pass: templateIdForTokenType returns non-existent template ID
- Bug: `templateIdForTokenType()` in GraphPanel.tsx returned `'modular-type-scale'` for non-color/dimension tokens, but the actual template ID in `GRAPH_TEMPLATES` is `'modular-type'`. This caused `GRAPH_TEMPLATES.find()` to return `undefined`, so the fallback picked `GRAPH_TEMPLATES[0]` (material-color) instead of the correct modular-type template when navigating to the Graph tab from a token's context menu.
- Fix: Changed the return value from `'modular-type-scale'` to `'modular-type'` to match the actual template ID.
- Files changed: `packages/figma-plugin/src/ui/components/GraphPanel.tsx` ~L613
- **Root cause:** Template ID string was out of sync with the template definition — likely a typo or rename that wasn't updated in the mapping function.

## 2026-03-27 - ux-pass: BatchEditor silent failures on Move, Rename, and Apply operations
- All three batch operations (Apply, Move, Rename) in BatchEditor.tsx used `fetch()` without checking `res.ok`. Since `fetch` only rejects on network errors (not HTTP 4xx/5xx), server-side failures were silently treated as successes — the UI showed "Moved N tokens" or "Renamed N tokens" even when the server rejected the requests.
- Changed all three handlers to validate HTTP response status and report partial failures accurately:
  - `handleApply`: uses `Promise.allSettled` + counts succeeded/failed, shows "N updated, M failed" on partial failure
  - `handleMove`: same pattern, only clears target set dropdown on full success
  - `handleRename`: checks `res.ok` per-iteration in the sequential loop, accumulates succeeded/failed counts
  - `patchToken` helper: now throws on non-2xx responses
- Files changed: `packages/figma-plugin/src/ui/components/BatchEditor.tsx` ~L115 (patchToken), ~L172 (handleApply), ~L208 (handleMove), ~L240 (handleRename)
- **User impact:** Users now see accurate feedback when batch operations partially or fully fail, instead of false success messages. This prevents the confusing scenario where tokens appear moved/renamed in the UI but remain unchanged on the server.

## 2026-03-27 - No way to identify and clean up derived tokens after deleting a generator
- Added `findTokensByGeneratorId(id)` and `deleteTokensByGeneratorId(id)` methods to `TokenStore` to scan flat token map for `$extensions["com.tokenmanager.generator"].generatorId` matches. Supports `'*'` wildcard to find all tagged tokens.
- Added three new API endpoints:
  - `GET /api/generators/:id/tokens` — list tokens created by a specific generator (works even after generator deletion by scanning token metadata)
  - `GET /api/generators/orphaned-tokens` — find all tokens whose generatorId doesn't match any existing generator
  - `DELETE /api/generators/orphaned-tokens` — bulk-delete all orphaned generator tokens
- Enhanced `DELETE /api/generators/:id` with `?deleteTokens=true` query param to optionally clean up derived tokens in one request.
- Updated TokenGraph UI `handleDelete`: after confirming generator deletion, fetches derived token count and shows a second confirm dialog offering to delete them. User can choose to keep or remove orphaned tokens.
- Files changed:
  - `packages/server/src/services/token-store.ts` ~L494 (findTokensByGeneratorId, deleteTokensByGeneratorId)
  - `packages/server/src/routes/generators.ts` ~L43 (orphaned-tokens GET/DELETE), ~L138 (/:id/tokens GET), ~L144 (DELETE with deleteTokens query)
  - `packages/figma-plugin/src/ui/components/TokenGraph.tsx` ~L121 (handleDelete with token cleanup flow)
- **Learnings for future iterations:**
  - Fastify static routes (`/generators/orphaned-tokens`) must be registered before parameterized (`/generators/:id`) — same registration-order principle as documented for token routes.
  - `window.confirm` two-step pattern (confirm action → confirm side effect) is a clean UX for "delete X and optionally Y" without needing a custom modal.

## 2026-03-27 - Duplicate targetGroup values across generators corrupt dependency resolution
- Changed `producerByGroup` in `buildDependencyOrder()` from `Map<string, string>` to `Map<string, Set<string>>` so that multiple generators sharing the same `targetGroup` are all registered as producers.
- Updated the edge-building loop to iterate over all producer IDs per group instead of a single one, ensuring correct dependency edges and topological ordering.
- Files changed: `packages/server/src/services/generator-service.ts` ~L225-L253
- **Learnings for future iterations:**
  - When building reverse-lookup maps (value → key), always consider whether the mapping is 1:1 or 1:N. Using `Map.set` for 1:N silently drops earlier entries — use `Map<K, Set<V>>` instead.

## 2026-03-27 - Generator route accepts unknown `type` without input validation
- Added `VALID_GENERATOR_TYPES` array and early validation in both POST `/generators` (create) and POST `/generators/preview` routes. Unknown type values now return 400 Bad Request with a descriptive message listing valid types, instead of falling through to the service layer's `default: throw` which returned a generic 500 Internal Server Error.
- Files changed: `packages/server/src/routes/generators.ts` ~L3 (constant), ~L84 (create validation), ~L115 (preview validation)
- **Learnings for future iterations:**
  - When a TypeScript union type is used as a route parameter but arrives as `string` at runtime, always validate before casting with `as`. The TS compiler can't catch invalid casts from `string` to a union, so runtime validation is the only defense.

## 2026-03-27 - Multi-brand mode silently skips the semantic mapping dialog
- In multi-brand mode, `fetchPreview` (line 162) immediately clears `previewTokens` to `[]` and returns without calling the preview API. The `handleSave` condition at line 292 (`previewTokens.length > 0`) therefore never triggered the `SemanticMappingDialog`.
- Fix: After a successful multi-brand save, fetch the generated tokens from the server via `GET /api/generators/:id/tokens` (the POST response body contains the generator ID, and the server runs the generator immediately on creation). Use those tokens to populate the semantic mapping dialog.
- Files changed: `packages/figma-plugin/src/ui/hooks/useGeneratorDialog.ts` ~L292
- **Learnings for future iterations:**
  - When multi-brand mode skips a live-preview step, any post-save flow that depends on preview data needs a fallback fetch from the server. The generator is already run on creation (server line 106-107), so `GET /generators/:id/tokens` is available immediately.
  - `fetch` Response body can only be consumed once — safe to call `res.json()` on the success path only if the error path (which also calls `res.json()`) returns early.
