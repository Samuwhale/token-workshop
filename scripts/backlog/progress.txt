## 2026-03-29 - Consistency scanner has quadratic time complexity on large pages
- Replaced O(nodes × tokens) linear scan with reverse-index lookups built once upfront:
  - **Color tokens**: Spatial hash (3D grid with bucket size = COLOR_NEAR_MAX=10) enables O(1) avg lookup by checking only 27 neighboring buckets instead of all color tokens
  - **Dimension tokens**: Sorted array + binary search for range queries → O(log n) per lookup
  - **Number/opacity tokens**: Same sorted-array approach → O(log n) per lookup
- Overall complexity reduced from O(nodes × tokens) to O(nodes × log(tokens)) for numeric properties and O(nodes × k) for colors where k is the avg bucket population (typically 1-3 tokens)
- Files changed: `packages/figma-plugin/src/plugin/consistencyScanner.ts` ~L36-128 (new index builders + query functions), ~L130-278 (refactored scanConsistency to use indexes)
- **Learnings for future iterations:**
  - The existing code already pre-bucketed tokens by type (color/dim/number) but still did linear scan within each bucket
  - `parseHexRaw` returns 0-1 RGB values, must scale to 0-255 for Euclidean distance — this is already noted in the original code but easy to forget
  - Spatial hashing with 3×3×3 neighbor search is a good general pattern for near-match color lookups with small thresholds

## 2026-03-29 - resolver-store.ts write guard starts BEFORE the actual file write
- Switched both `resolver-store.ts` and `token-store.ts` to atomic writes: write JSON to `.tmp` file first, then `fs.rename` to final path. `fs.rename` is atomic on the same filesystem, so the file watcher never sees a partially-written file.
- Write guard now starts just before the (instantaneous) rename, not before the (slow) writeFile — eliminates the race window where the guard could expire during a long write.
- Increased resolver-store guard timeout from 500ms to 30s (matching token-store) — the short timeout was a secondary bug that could silently drop external edits.
- Added `_clearWriteGuard` to resolver-store and updated `onFileChange`/`onFileRemove` to clear guards immediately on watcher callback (matching token-store's pattern), instead of relying solely on timeouts.
- Files changed: `packages/server/src/services/resolver-store.ts` ~L202-230, ~L272-283; `packages/server/src/services/token-store.ts` ~L511-519, ~L1647-1658
- **Learnings for future iterations:**
  - Atomic write (tmp + rename) is strictly better than write guards around `writeFile` — it eliminates partial reads entirely and makes the guard window negligible
  - `resolver-store.ts` was missing `_clearWriteGuard` that `token-store.ts` already had — when fixing a pattern in one store, check the other store for the same issue
