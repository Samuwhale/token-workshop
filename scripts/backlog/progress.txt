## 2026-03-28 - useFigmaSync.ts error handlers restore stale closure values
- Fixed stale closure bug in `handleSyncGroup` and `handleSyncGroupStyles` error handlers
- Both catch blocks were calling `setSyncGroupPending(syncGroupPending)` / `setSyncGroupStylesPending(syncGroupStylesPending)` which captured the state value at `useCallback` creation time. On error after the state was cleared to `null`, this would set it back to whatever stale value was in the closure — potentially `null` itself or an outdated value from a previous render
- Fix: save the pending value to a local `const saved` before clearing to `null`, then restore `saved` on error. Also used `saved.groupPath` instead of destructured `groupPath` for consistency
- Files changed: `packages/figma-plugin/src/ui/hooks/useFigmaSync.ts` ~L22-41, ~L43-62
- **Learnings for future iterations:**
  - When a `useCallback` clears state then needs to restore it on error, always capture the value in a local variable before clearing — the closure variable may be stale by the time the catch block runs
  - The `syncGroupPending` state was already in the `useCallback` deps array, so the closure value matched the render-time value, but the real issue is that between `setSyncGroupPending(null)` and the catch block, a re-render may have already happened with the null value — the original closure variable still holds the old value, but writing `setSyncGroupPending(syncGroupPending)` is semantically wrong (it's a no-op or stale restore). Using a saved local makes the intent explicit.

## 2026-03-28 - Plugin sandbox operations don't use figma.group() for undo batching
- Removed `setTimeout(resolve, 0)` yield between batches in `syncBindings()` — this was breaking Figma's automatic undo grouping, causing each 50-node batch to become a separate undo step
- Added explanatory comment documenting why `setTimeout` must not be used between mutation batches
- Other batch operations (`applyVariables`, `applyStyles`, `applyToSelection`, `batchBindHeatmapNodes`) already operate within a single undo group since they don't use `setTimeout` — only Figma async API awaits which preserve undo context
- The `setTimeout` yields in `heatmapScanning.ts` (`scanCanvasHeatmap`, `scanTokenUsage`) are in read-only scanning functions, so they don't affect undo grouping
- Files changed: `packages/figma-plugin/src/plugin/selectionHandling.ts` ~L784
- **Learnings for future iterations:**
  - Figma Plugin API groups all mutations within a single message handler as one undo step by default
  - `setTimeout(resolve, 0)` breaks this automatic grouping — each setTimeout continuation becomes a new undo step
  - `await` on Figma async APIs (`loadFontAsync`, `getNodeByIdAsync`, etc.) does NOT break undo grouping — they yield to the runtime but stay in the same undo context
  - `figma.commitUndo()` SPLITS operations into separate undo steps — it does NOT group them. There is no explicit "group as undo" API; the default behavior is already one undo step per handler
  - The backlog item referenced `figma.group()` but that's for visual node grouping, not undo batching
