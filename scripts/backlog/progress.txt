  - React error #300 ("Cannot update a component while rendering a different component") can be triggered by unstable function/object props that cause downstream `useCallback` to recreate on every render and setState during render indirectly. Root cause is always a hook or callback defined outside of hook scope (inside a conditional/return branch).

## 2026-04-04 - Manual snapshot recovery retries the same failed set infinitely on every server startup
- Added `failedSets?: Record<string, number>` to `RestoreJournal` to track per-set retry counts. In `recoverPendingRestore`, sets at or above `MAX_RECOVERY_RETRIES` (3) are quarantined with a console.error and skipped. Sets below the threshold are attempted; on failure the counter is incremented and persisted to the journal. Recovery now iterates all pending sets instead of returning on first failure (`allResolved` flag). The journal is deleted only when all pending sets are either completed or quarantined.
- Files changed: `packages/server/src/services/manual-snapshot.ts` ~L17-24 (interface), ~L51 (constant), ~L308-345 (recoverPendingRestore loop)
- **Learnings for future iterations:**
  - Quarantined sets use `continue` (not `return`) so other pending sets still get their chance — important to process all sets in one startup pass.
  - Writing the updated `failedSets` count back to the journal immediately after each failure ensures the counter survives another crash mid-recovery.
  - `allResolved` starts `true` and flips `false` only on an actual failure (not on quarantine) — quarantined sets are intentionally "resolved" so they don't block journal cleanup indefinitely.

  2. `_doSave()` now wraps `fs.rename` in try/finally to unlink the `.tmp` file on rename failure, preventing temp file leaks.
  3. `GET /generators` route now uses `stableStringify` (imported from `../services/stable-stringify.js`) instead of `JSON.stringify` for the `isStale` comparison, eliminating false-positive staleness for objects with non-deterministic key ordering.
- Files changed: `packages/server/src/services/generator-service.ts` ~L116, ~L141, ~L168, ~L183; `packages/server/src/routes/generators.ts` ~L22, ~L392
- **Learnings for future iterations:**
  - Mutation-before-save pattern is easy to overlook: whenever you mutate in-memory state and then await a disk write, add a try/catch that restores the prior state on failure.
  - `.tmp` file rename failures require explicit cleanup — the same `try { rename } catch { unlink(tmp); throw }` pattern applies anywhere atomic writes are used.
  - Any staleness/comparison logic that operates on nested objects must use `stableStringify`, not `JSON.stringify` — applies to both service and route layers.
